1. CompositeRoomStore: Durable / Realtime 分层1. CompositeRoomStore: Durable / Realtime Layering

CompositeRoomStore 定义了后端的状态归属：哪些数据必须持久，哪些状态可以重建，以及客户端最终以哪一层为准。它组合了三个职责不同的子存储：

CompositeRoomStore defines state ownership in the backend: which data must persist, which state can be rebuilt, and which authority clients eventually follow. It composes three stores with different responsibilities:

  • DurableRoomStore：PostgreSQL 是唯一 runtime 实现，保存房间、消息、room events、成员、媒体 metadata、认证、assistant_runs 与 dispatch intent。旧 RedisStore 只为 import/migration 与 contract coverage 保留。
  • RealtimeRoomStore：始终是 Redis。管理 socket session、在线成员和临时状态；同一个 Redis 初期也以独立 namespace/connection 承载 BullMQ。
  • RoomMessageCacheStore：可选的 Redis TTL 缓存，在 PostgreSQL 模式下减少数据库读压力。
  • DurableRoomStore: PostgreSQL is the only runtime implementation, owning rooms, messages, room events, members, media metadata, auth, assistant_runs, and dispatch intent. The legacy RedisStore remains only for import/migration and contract coverage.
  • RealtimeRoomStore: Always Redis. It manages socket sessions, online members, and ephemeral state; the same Redis initially hosts BullMQ through separate namespaces and connections.
  • RoomMessageCacheStore: Optional Redis TTL cache that reduces database read pressure in PostgreSQL mode.
面试时，我会先说明状态归属。房间、消息、event sequence 和 AI 业务结果无法从运行队列反推，所以由 PostgreSQL 保存；session、presence 与短 TTL cache 可以重建。BullMQ job 也放在 Redis，但它只描述“如何调度”，不回答用户任务是否成功；active job 用 AOF 保存，不能当普通 cache 清空。客户端按房间 sequence 重放，Socket.IO 与 BullMQ 都不是最终业务事实源。
In an interview, I start with state ownership. Rooms, messages, event sequences, and AI business outcomes cannot be inferred from a runtime queue, so PostgreSQL stores them. Sessions, presence, and short-TTL cache can be rebuilt. BullMQ jobs also live in Redis, but describe scheduling rather than whether the user's task succeeded; active jobs use AOF and are not flushed like cache. Clients replay room sequences, while neither Socket.IO nor BullMQ is the final business authority.

2. Socket.IO 多实例扩展2. Socket.IO Multi-Instance Scaling

durable room event: PostgreSQL NOTIFY → 每个 App → io.local → 本机客户端 single-origin transient event: App A → Redis adapter → App B/C → 对应客户端
durable room event: PostgreSQL NOTIFY → every app → io.local → attached clients single-origin transient event: App A → Redis adapter → App B/C → target clients

多实例投递有两条路径。PostgreSQL NOTIFY 已经把 durable room event 发送给每个监听实例，因此每个实例读取精确事件行后只做 io.local 广播。若再经过 Redis adapter，同一事件会被重复放大。由单个实例产生的 transient/global event 才使用 @socket.io/redis-adapter。客户端无须关心连接落在哪个实例上，服务端则要根据事件来源选择投递路径。

Multi-instance delivery has two paths. PostgreSQL NOTIFY already fans a durable room event out to every listening app, so each app reads the exact event row and broadcasts with io.local; sending it through the Redis adapter would multiply the same event. The @socket.io/redis-adapter is for genuinely single-origin transient or global events. Clients do not need to know which instance owns their connection, but the server must choose the path that matches the event's origin.

这里还要区分“收到 NOTIFY”和“本机需要读 payload”。A、B、C 都会收到数据库 hint，但只有本机真的连接了该 room socket 的实例才读取事件行；其余实例直接跳过。通知突发时，每个房间只保存最低和最高 pending seq，不给每条通知创建 Promise waiter。发完整 payload 前，实例以本机 socket.data 为身份权威，再用一次 PostgreSQL 查询判断成员。Redis 索引缺失可由本机身份重建；本机身份缺失或与 Redis 冲突时，只让异常 socket 重新注册并离开 room，不影响其他连接。明确 unauthorized 才移出 room；PostgreSQL 授权查询 unavailable 时保留连接，只发 head-only，让后续 GET 再鉴权。这样依赖短暂故障不会伪装成“你被踢出房间”。

Receiving NOTIFY is separate from needing the payload locally. A, B, and C all receive the database hint, but only an instance with a local socket in that room reads the event row. Burst notifications keep one fixed min/max pending range per room rather than one Promise waiter per notification. Before a complete payload leaves the instance, local socket.data is treated as the identity authority and PostgreSQL membership is queried once. A missing Redis index is rebuilt from local identity. A socket with no local identity, or a Redis identity that conflicts with it, is individually asked to register and removed from the room without affecting verified peers. Only explicit non-members are otherwise removed. If PostgreSQL authorization is unavailable, connections stay joined and receive only a head hint so the later GET can authorize again. A transient dependency outage therefore cannot masquerade as a membership revocation.

因此滚动启动不再执行“清空全部 presence、把全部 running 标成 error”。每个进程生成唯一 instance ID 并续租；Redis Lua 清理会重新检查 heartbeat，只处理仍属于过期 owner 的 socket。Code Agent recovery 会排除未过期 fenced lease，AI recovery 会排除未过期 stream owner。所有副本都可以运行同一套维护 loop，但只有拿到 advisory lock 的实例真正执行。这是从单机可运行走向 ECS/EKS 滚动发布必须补齐的 ownership contract。不过首次引入 0006、0007 和 0008 时仍要停止旧 App/Worker：旧 binary 既不会写 owner lease，也不会用 generation fence 条件提交 AI 终态。所有副本跨过这条边界后，后续协议兼容版本才适合滚动。

Rolling startup therefore no longer means clearing all presence or marking every running task failed. Each process gets a unique instance ID and renews its lease; Redis Lua cleanup rechecks the heartbeat and touches only sockets still owned by the expired instance. Code Agent recovery excludes unexpired fenced leases, and AI recovery excludes unexpired stream owners. Every replica may run the same maintenance loop, but only the advisory-lock holder performs a pass. This ownership contract is what turns a single-host design into one that can safely roll on ECS or EKS. The first 0006/0007/0008 deployment is still a stop-the-world boundary: old binaries neither write owner leases nor condition terminal AI writes on a generation fence. Later protocol-compatible releases may roll after every replica has crossed that boundary.

追问：sticky session 为什么必须？

如果允许 polling fallback，Socket.IO 握手会先做 HTTP polling 再升级 WebSocket；这些请求落到不同实例会破坏 session，因此负载均衡需要 connection affinity。也可以强制 websocket-only 来减少这个要求，但会放弃 polling fallback。Sticky session 解决连接归属，不决定 durable event 的 fan-out；后者仍由 PostgreSQL NOTIFY + io.local 完成。

Follow-up: Why are sticky sessions required?

With polling fallback enabled, Socket.IO starts with HTTP polling and upgrades to WebSocket. If those requests land on different instances, the session breaks, so the load balancer needs connection affinity. WebSocket-only transport can reduce this requirement at the cost of losing fallback. Sticky sessions solve connection ownership; they do not define durable-event fan-out, which uses PostgreSQL NOTIFY plus io.local.

  • 一个 PostgreSQL 事务同时创建 status: streaming placeholder、assistant_runs、placeholder room event 与 task_dispatch_outbox。业务请求与待投递意图要么一起提交，要么一起回滚。
  • App relay 只把 { schemaVersion: 1, runId } 以 jobId=runId 送入 BullMQ。Redis 暂时不可用时 dispatch row 保持 pending，所以“用户看到已接受但任务消失”的窗口被关闭。
  • Dispatch 已确认后仍有周期对账：App 在 PostgreSQL advisory lock 下只读取未终态 run。BullMQ job 缺失时按同一 runId 补建；failed 或业务仍 active 却提前 completed 的 job 重新放回 waiting；waiting/active/delayed job 不动。
  • 独立 Node/TypeScript Worker 从 PostgreSQL 读取 request snapshot，claim 精确 run 并得到 generation lease。BullMQ 负责 waiting、并发、backoff 与 stalled recovery；assistant_runs 才负责业务状态和结果。
  • Worker 通过版本化 Redis channel 发布 ai_chunk/a2ui_update/ai_stream_end；每个 App 重新鉴权本机 socket 后用 io.local 转发。早到事件按 messageId 有界暂存，旧 generation 被忽略。
  • Provider 完成后先把 immutable terminal payload 写入 run，再在一个事务里投影最终 Message、run 状态与房间费用。Terminal staging 之后崩溃，retry 只重做 projection。若进程在 Provider 接受请求后、staging 前退出，外部调用仍是至少一次；generation fence 只接受一个终态，内部费用也只结算一次。
  • 终态只允许更新同 room、仍为 streaming、owner/generation 匹配的现有 placeholder。消息已删除、已完成或被新 Worker 接管时，旧 job 变 obsolete，绝不会 upsert 复活。
  • One PostgreSQL transaction creates the status: streaming placeholder, assistant_runs, placeholder room event, and task_dispatch_outbox. The accepted business request and enqueue intent commit or roll back together.
  • The App relay sends only { schemaVersion: 1, runId } to BullMQ with jobId=runId. If Redis is temporarily unavailable, the dispatch row stays pending, closing the “accepted request silently vanished” window.
  • Acknowledged dispatch is still reconciled. Under a PostgreSQL advisory lock, the App checks only non-terminal runs: it recreates a missing deterministic job, retries failed or prematurely completed jobs, and leaves waiting/active/delayed work alone.
  • A dedicated Node/TypeScript Worker reads the request snapshot from PostgreSQL and claims the exact run with a generation lease. BullMQ owns waiting, concurrency, backoff, and stalled recovery; assistant_runs owns business status and results.
  • The Worker publishes ai_chunk/a2ui_update/ai_stream_end through a versioned Redis channel. Every App reauthorizes attached sockets and emits with io.local. Early events are bounded-buffered by messageId; older generations are ignored.
  • After the Provider completes, the Worker first stages an immutable terminal payload, then projects final Message, run state, and room cost in one transaction. A crash after staging retries projection only. A crash after remote acceptance but before staging can repeat the external call; generation fencing still admits one terminal result and settles RoomTalk's internal cost once.
  • Terminal projection updates only an existing placeholder in the same room that is still streaming and owned by that generation. Deleted, terminal, or replacement-owned messages make the old job obsolete rather than being upserted back to life.

追问：为什么不用 SSE？

SSE 可以承载单向的 AI token 流。RoomTalk 已经用 Socket.IO 处理房间事件、ack、重连和身份上下文，复用这条双向连接可以少维护一套鉴权、断线恢复和流生命周期。这个选择来自现有系统边界。如果服务只需要 HTTP 请求和单向响应流，SSE 会更简单。

Follow-up: Why not use SSE?

SSE can carry a one-way AI token stream; there is no technical incompatibility. RoomTalk already uses Socket.IO for room events, acknowledgments, reconnect recovery, and identity context, so reusing that bidirectional connection avoids maintaining a second authentication and stream-lifecycle path. This is a tradeoff inside the existing system boundary, not a claim that SSE is unsuitable for AI streaming. For a service that only needs an HTTP request and a one-way response stream, SSE may be simpler.

The signature binds the HTTP method, bucket, object key, and expiry; an upload URL cannot read unrelated objects. The current SeaweedFS endpoint is used only through the controlled S3-compatible boundary, not through a public object listing. A presigned URL is still a short-lived bearer capability, so full query strings must stay out of logs, keys must be unguessable, and the server must recheck room access both before signing and at confirmation.

  • Model gateway：每个 turn 签发绑定 room/client/turn/provider/model/budget 的短期 token；provider key 不进入浏览器，也不作为 prompt 内容。
  • Room context broker：不把整个房间历史塞进 system prompt；Agent 通过 Unix socket 上的 roomtalk room history/search/delta/message 按需读取有界 projection。
  • Static publish：可写模式拿到独立 publish token；文件从 E2B 通过 presigned PUT 直传当前 SeaweedFS/S3-compatible store，RoomTalk 只验证 metadata 并原子切换 manifest。
  • Four modes: Plan, Edit, Approve for me, and Full access. Plan shell writes are blocked by the OS mount namespace, not a string command allowlist.
  • Model gateway: each turn receives a short-lived token bound to room/client/turn/provider/model/budget; provider keys never enter the browser or prompt.
  • Room-context broker: full room history is not injected into every system prompt. The agent reads bounded projections on demand through roomtalk room history/search/delta/message over a Unix socket.
  • Static publishing: writable modes receive a separate publish token. E2B uploads bytes directly to the current SeaweedFS/S3-compatible store through presigned PUTs; RoomTalk verifies metadata and atomically swaps the manifest.

追问：为什么不用一个万能 token？

因为 context read、model spend 和 public publish 是三种不同权限：历史 token 只读当前房间的安全 projection，model token 只允许本轮选定的 provider/model 和预算，publish token 只允许当前房间发布 artifact。拆成独立、短期、绑定 turn 的 capability 后，其中一个泄露也不能替代另外两个，服务端还能分别校验 mode、room ownership、provider/model 和 budget。

Follow-up: Why not issue one universal token?

Context reads, model spending, and public publishing are different permissions. A context token can only read a safe projection of the current room; a model token is restricted to the selected provider/model and turn budget; a publish token can only publish artifacts owned by the room. Separate short-lived, turn-bound capabilities are non-interchangeable and let the server independently enforce mode, room ownership, provider/model selection, and budget.

Solution:

The registry contains live stdin/stdout handles owned by the current Node process, which cannot simply be serialized. Startup cleanup, health checks, and process-exit reclamation manage ownership, while room and sandbox identity remain durable. 追问：既然 Socket 能带 payload，为什么还保留 GET？

因为 Socket fast path 优化的是常见延迟，不承担唯一正确性。正常收到连续的 { headSeq: 1042, events: [1042] } 时会直接应用，省掉图中的第一个 GET；但客户端可能离线、通知可能重复/乱序、payload 可能超过 256 KiB，精确 event read 也可能短暂失败。headSeq 告诉客户端目标水位，PostgreSQL replay 才能证明 1042 到 1045 一条不缺。不能证明连续时就补拉或 snapshot。

Follow-up: If Socket can carry payload, why keep GET?

The Socket fast path optimizes common-case latency; it is not the sole correctness path. A normal contiguous { headSeq: 1042, events: [1042] } is applied directly and removes the first GET shown above. But a client may be offline, notifications may duplicate or reorder, a payload may exceed 256 KiB, or the exact-event read may briefly fail. headSeq gives the target high-water mark, while PostgreSQL replay proves that nothing from 1042 through 1045 is missing. If continuity cannot be proven, the client replays or resnapshots.

数据库恢复后，为什么要清掉旧水位？

假设浏览器缓存到 lastAppliedSeq=10，旧通知还要求追到 20，但数据库从备份恢复后只有 8。若 snapshot 把 cursor 设为 8，却继续保留目标 20，客户端会不断请求 afterSeq=8，每次都得到空页和 headSeq=8。所以处理 CURSOR_AHEAD 时先把旧目标清零。若 snapshot 请求期间数据库真的提交了 9，新的 Socket 通知会把目标重新设为 9；snapshot 应用时用 max 保留这个新水位，随后再拉 9。被丢弃的是恢复前已经不存在的目标，不是恢复后的新事件。

Why clear the old watermark after a database restore?

Suppose the browser cached lastAppliedSeq=10 and an old notification still targets 20, while the restored database ends at 8. If a snapshot moves the cursor to 8 but retains target 20, the client repeatedly asks for afterSeq=8 and receives the same empty page with headSeq=8. On CURSOR_AHEAD, it therefore clears the obsolete target first. If the database commits 9 while the snapshot request is running, the new Socket notification raises the target to 9; snapshot application retains that new maximum, then fetches 9. Recovery discards a target that no longer exists, not events committed after recovery.

为什么这里必须是状态机，而不是继续加几个 boolean？

replay、replace 和 prepend 都会异步返回消息数组，但语义不同：replay 推进 durable cursor，replace 重建规范窗口，prepend 只给 UI 补更早历史。过去它们共享一个 sequence counter，用户滚动触发 prepend 时可能把正在恢复的 replace response 判成过期。现在 controller 明确规定优先级：replace 可以使旧 prepend 失效，prepend 却不能进入正在 replay/replace 的房间；被拒绝的分页只是稍后重试，不会取消正确性恢复。状态机还统一拥有 desiredHeadSeq、gap target 和 history-invalidated 标记，所以数据库恢复、删空窗口与新通知交错时都有唯一转移规则。

Why does this need a state machine instead of a few more booleans?

replay, replace, and prepend all return message arrays asynchronously, but they mean different things: replay advances the durable cursor, replace rebuilds the canonical window, and prepend adds older UI history only. They previously shared one sequence counter, so scrolling could make an in-flight recovery response look stale. The controller now makes priority explicit: replace may invalidate an older prepend, while prepend cannot enter a room already replaying or replacing. Deferred pagination can retry later without cancelling correctness recovery. The same controller owns desiredHeadSeq, the gap target, and history invalidation, giving database restore, emptied-window deletion, and new notification interleavings one transition model.

Canonical tables 仍是事实源，room_events 是 fan-out after-image changelog：每个客户端用自己的 cursor 重读。AI 调度是 competing-consumer：task_dispatch_outbox 只桥接 PostgreSQL commit 与 Redis，BullMQ job 只由一个 Worker 执行，而 assistant_runs 保存业务结果。三者都可能与同一次用户操作有关，但 consumer、重试与 retention 完全不同，所以不能合并成一张万能事件表。
Canonical tables remain authoritative, while room_events is a fan-out after-image changelog that every client reads with its own cursor. AI scheduling is competing-consumer work: task_dispatch_outbox only bridges the PostgreSQL commit into Redis, one Worker executes a BullMQ job, and assistant_runs stores the business result. These mechanisms may originate in one user action, but their consumers, retries, and retention differ, so one universal event table would blur rather than simplify the model.

追问：为什么不用 CRDT？

房间设置是低频的完整对象更新，last-write-wins 已经够用。数据库 trigger 保证 updatedAt 在串行化写入中严格单调，用作完整对象的 ack/broadcast guard；room.updated 同时进入 durable event cursor，漏掉广播也能恢复。CRDT 更适合高频的字段级协同编辑，在这里会增加协议和调试成本，却没有对应收益。

Follow-up: Why not use CRDTs?

Room settings are low-frequency complete-object updates, so last-write-wins is sufficient. A database trigger makes updatedAt strictly monotonic across serialized writes; it is only the complete-object ack/broadcast guard. room.updated also enters the durable event cursor, so missed broadcasts recover. CRDT complexity is better suited to high-frequency field-level collaborative editing.

  • 本地 cacheGeneration 只是跨 tab invalidate 的 browser epoch，不是业务同步 version。
  • desiredHeadSeq 只维护收到的最高水位，多个通知合并为一次追赶；差距超过 500 直接 snapshot。CURSOR_AHEAD 会清除恢复前的旧目标，但保留 snapshot 进行中收到的新目标。
  • Redis 最近消息 cache 同样可丢；异常直接降级读 PostgreSQL。
  • room.deleted 同时写 tombstone、清 message/media cache；晚到 callback 不能复活已删除房间。
  • The memory window paints instantly within a tab; IndexedDB v4 restores across reloads.
  • Local cacheGeneration is only a cross-tab invalidation epoch, not a business synchronization version.
  • desiredHeadSeq retains only the highest notified watermark and coalesces many notifications into one catch-up. A gap over 500 resnapshots. CURSOR_AHEAD clears the obsolete pre-restore target while preserving a new target received during the snapshot.
  • The Redis recent-message cache is also disposable; failures fall back to PostgreSQL.
  • room.deleted records a tombstone and clears message/media caches so late callbacks cannot resurrect the room.
退役 messageVersion 后，一致性边界更清楚：服务端只有 per-room event sequence，客户端只推进连续的 lastAppliedSeq。所有 cache 都是带 cursor 的性能优化，不是事实源；无法证明连续就 snapshot，而不是猜测或 merge。
After retiring messageVersion, the consistency boundary is clearer: the server exposes one per-room event sequence and the client advances only a contiguous lastAppliedSeq. Every cache is a cursor-tagged performance optimization, never a source of truth; if continuity cannot be proven, resnapshot instead of guessing or merging.

AI 上下文预算管理（CJK 感知）CJK-Aware AI Context Budgeting

问题：AI 请求附带聊天历史上下文，太长会超过模型 token 限制。这里采用可解释的保守估算：CJK 约每个字符 1 token，非 CJK 文本约每 4 个字符 1 token，混合内容分别计算。

Problem: AI requests include chat history, and an oversized context can exceed the model's token limit. The implementation uses an explainable conservative estimate: roughly one token per CJK character and one per four non-CJK characters, calculated separately for mixed text.

// aiHistory.ts: CJK 感知的 token 估算
const estimateTextTokens = (text: string): number => {
  const cjkCount = text.match(CJK_CHARACTER_PATTERN)?.length ?? 0;
  const nonCjkText = text.replace(CJK_CHARACTER_PATTERN, '');
  return Math.max(1, cjkCount + Math.ceil(nonCjkText.length / 4));
  // CJK: 1 字符 ≈ 1 token；拉丁: 4 字符 ≈ 1 token
};

三层筛选（从粗到细）：

Three-Layer Selection (coarse to fine):

// aiHistory.ts: selectAIHistory
export function selectAIHistory(fullHistory, options) {
  let history = fullHistory;

  // 第 1 层：语义截断（重试/编辑时截到该消息之前）
  if (retryForMessageId) {
    const idx = history.findIndex(m => m.id === retryForMessageId);
    if (idx !== -1) history = history.slice(0, idx);
  }

  // 第 2 层：消息条数限制（最近 N 条）
  if (history.length > maxContextMessages)
    history = history.slice(-maxContextMessages);

  // 第 3 层：token 预算（从最新往前贪心累加）
  const selected = [];
  let tokenCount = 0;
  for (let i = history.length - 1; i >= 0; i--) {
    const msgTokens = estimateMessageTokens(history[i]);
    if (selected.length > 0 && tokenCount + msgTokens > maxContextTokens)
      break;
    selected.push(history[i]);
    tokenCount += msgTokens;
  }
  return { contextMessages: selected.reverse(), contextTokenEstimate: tokenCount };
}
AI 上下文会经过三次筛选。用户重试或编辑消息时，先按语义截断，只保留目标消息之前的历史；然后限制消息条数；最后从最新消息开始向前累加，直到用完 token 预算。估算时分开计算 CJK 和拉丁字符：中文约每个字 1 token，英文约每 4 个字符 1 token。这个估算偏保守，宁可少带一条消息，也不让请求超限。
AI context passes through three filters. A retry or edit first truncates history at the target message. The selector then applies the message-count limit and walks backward from the newest message until it reaches the token budget. CJK and Latin text are estimated separately: about one token per CJK character and one per four Latin characters. The estimate is deliberately conservative; leaving out one message is safer than exceeding the model limit.

The clientId is bound to an auth token (stored as SHA-256 hash). Socket registration requires both clientId and token: mismatched tokens are rejected. 为什么用 E2B，而不是在 RoomTalk 应用容器里直接跑 Docker 或 Shell？
RoomTalk 进程拥有数据库、Redis、对象存储和用户 session 等生产权限，不应该同时执行模型生成的不可信命令。E2B 提供独立生命周期、文件系统、进程和网络边界。RoomTalk 只通过窄协议控制 sandbox，即使 Agent 命令失控，也不会直接进入 control plane。
Why E2B instead of running Docker or shell directly in the RoomTalk application container?
The RoomTalk process owns production database, Redis, object-storage, and user-session privileges, so it must not also execute model-generated commands. E2B provides an independent filesystem, process, network, and lifecycle boundary. RoomTalk controls it through a narrow protocol, keeping failures out of the control plane.
为什么需要 sandbox daemon，而不是每轮启动一次 runner？
每轮进程适合 MVP，但会重复加载 Python/Agent runtime，也很难拥有 terminal、后台服务和 backend thread 等跨 turn 状态。一个 sandbox 一个 daemon 可以复用执行环境，同时仍由 RoomTalk 保证同房间只跑一个 mutating turn。代价是必须处理 stale process、health check、shutdown ownership 和丢失 release signal。
Why use a sandbox daemon instead of starting one runner per turn?
Per-turn processes are useful for an MVP, but repeatedly load the runtime and cannot naturally own terminals, background services, or backend thread state across turns. One daemon per sandbox reuses the environment while RoomTalk still serializes mutating turns. The tradeoff is explicit stale-process cleanup, health checks, shutdown ownership, and lost-release handling.
多个成员同时给 Agent 发消息，怎么避免同时修改 workspace？
房间有一个 active-turn guard，真正执行的 mutating turn 串行化。后续 prompt 持久化为 queue，可以编辑、取消或提升为 steer。这样聊天仍是多人实时的，但共享文件系统不会被两个 Agent turn 并发修改。每个 turn 还持久化实际 mode/backend，retry 不会回退到默认权限。
How do you prevent multiple room members from modifying the workspace concurrently?
A room-level active-turn guard serializes mutating agent work. Later prompts become durable queue entries that can be edited, canceled, or promoted into steering input. Chat remains realtime and multi-user, while the shared filesystem never has two agent turns writing concurrently. Each turn persists its resolved mode/backend so retry cannot fall back to default permissions.
Agent 如何看到历史，又不会拿到整个数据库或长期 token？
Runner 为本轮创建 Unix socket broker，只把 roomtalk CLI 暴露给 Agent。CLI 的 history/search/delta/message 请求由 broker 带 scoped token 转发；服务端重新校验 room/client/turn，并只返回安全 projection。Model spend 与 static publish 使用另外两种 token，不能互相替代。
How can the agent read history without receiving database access or a long-lived token?
The runner creates a per-turn Unix-socket broker and exposes only the roomtalk CLI. The broker forwards history/search/delta/message requests with a scoped token; the server revalidates room/client/turn and returns a safe projection. Model spending and static publishing use separate non-interchangeable tokens.
Sandbox 暂停或升级后，文件和网页会怎样？
默认 idle 两分钟后 E2B pause，并启用 keep-memory + autoResume，所以同一个 sandbox 恢复时文件和进程内存可以继续。固定 artifact 版本变化时，RoomTalk 会 bounded export/import workspace archive 后原子替换 sandbox。公开静态站点单独存入当前 SeaweedFS/S3-compatible manifest/version，不依赖 sandbox 存活。
What happens to files and published pages when a sandbox pauses or upgrades?
The default E2B policy pauses after two idle minutes with keep-memory and autoResume, so the same sandbox can continue. When the pinned artifact changes, RoomTalk performs a bounded workspace archive export/import and atomically swaps sandboxes. Public static sites live in separate SeaweedFS/S3-compatible manifests and versions and do not depend on sandbox lifetime.
为什么用 Socket.IO 而不是原生 WebSocket？
Socket.IO 提供自动重连、房间/命名空间、ack 回调和可选 Redis adapter。原生 WebSocket 需要自己补这些协议层能力。RoomTalk 仍把正确性放在 PostgreSQL room-event cursor 上：durable event 由 PG NOTIFY 分发到每个 App，再用 io.local 通知；Redis adapter 只处理真正的单源 transient/global event。选择 Socket.IO 是为了连接与交互便利，不是把它当 durable queue。
Why Socket.IO instead of raw WebSocket?
Socket.IO provides reconnection, rooms/namespaces, acknowledgement callbacks, and an optional Redis adapter. Raw WebSocket would require those protocol features to be built separately. RoomTalk still places correctness in the PostgreSQL room-event cursor: PG NOTIFY reaches every app for durable events, followed by io.local; Redis adapter carries only genuinely single-origin transient or global events. Socket.IO is a connection and interaction convenience, not a durable queue.
为什么不用微服务架构？
RoomTalk 没有把每个领域都拆成远程服务，但也不再是单进程。App 与普通 Chat AI Worker 使用同一镜像和 TypeScript domain code，却以不同进程部署：App 拥有 HTTP/Socket 与 dispatch relay，Worker 拥有 BullMQ consumer 和 Provider 调用。它们通过 PostgreSQL aggregate 与最小 runId queue contract 连接，不互相调用内部 HTTP。这样 AI 并发、重启和故障域可独立控制，同时避免在 Python 或第二个服务里复制 Provider/A2UI/费用语义。E2B execution plane 则因为运行不可信代码，按安全边界真正隔离。
Why not a microservices architecture?
RoomTalk does not turn every domain into a remote service, but it is no longer one process either. App and ordinary chat AI Worker share one image and TypeScript domain code while deploying as separate processes: App owns HTTP/Socket and the dispatch relay; Worker owns the BullMQ consumer and Provider calls. They meet through the PostgreSQL aggregate and a minimal runId queue contract, not internal HTTP calls. This separates AI concurrency, restart, and failure behavior without copying provider/A2UI/cost semantics into Python or another service. The E2B execution plane is genuinely isolated because it runs untrusted code.
Redis 挂了怎么办？
需要区分 realtime Redis 与 queue 责任，即使它们当前是一个实例。业务事实仍在 PostgreSQL；cache/presence 丢失会重建，本机 Socket 继续以认证后的 socket.data 为身份权威。Queue 不可用时，App 已提交的 task_dispatch_outbox 保持 pending，状态接口显示 degraded/deferred，Redis 恢复后 relay 再送入 BullMQ。若 dispatch 已 ack 但 job 在 AOF 窗口或 volume 恢复中丢失，active-run reconciler 会按 PostgreSQL 重建；attempts 耗尽但 run 仍 active 也会重试。生产仍使用 named volume、AOF everysec 和 noeviction，因为保护现场比恢复更便宜；不能把 queue 当普通 cache 随手 flush。
What happens if Redis goes down?
Realtime and queue responsibilities must be separated conceptually even though they initially share one Redis. Business facts remain in PostgreSQL; cache/presence rebuild, and attached sockets retain authenticated socket.data identity. If the queue is unavailable, committed task_dispatch_outbox rows stay pending, status reports degraded/deferred dispatch, and the relay enqueues them after recovery. If an acknowledged job disappears in an AOF window or volume restore, active-run reconciliation rebuilds it from PostgreSQL; an exhausted job is also retried while its run remains active. Production still uses a named volume, AOF everysec, and noeviction because preserving live scheduling state is cheaper than recovery. Queue Redis is not casually flushed like cache.
room_event_available 有必要带 payload 吗？
有，作为常见路径优化：事务提交后，app 按 headSeq 从 PostgreSQL 读回已经保存的不可变 after-image，完整通知不超过 256 KiB 就直接带给客户端。只要它与 lastAppliedSeq 连续，客户端立即应用并省掉一次 GET。但 payload 不是唯一投递通路；超限、读取失败、离线、乱序或 gap 都用 headSeq 回到同一 durable log replay。这样兼顾低延迟与可证明恢复，也不会让旧 seq 被当前行污染。
Should room_event_available include a payload?
Yes, as a common-case optimization. After commit, the app reads the stored immutable after-image at headSeq from PostgreSQL and includes it when the complete notification is at most 256 KiB. If it is contiguous with lastAppliedSeq, the client applies it immediately and saves a GET. It is not the only delivery path: oversized events, read failures, offline clients, reordering, or gaps fall back to the same durable log using headSeq. This combines low latency with provable recovery without letting current rows corrupt old sequences.
为什么每个 App 都 LISTEN，却必须用 io.local？
PostgreSQL 已经把同一个 NOTIFY 送到 A、B、C 三个 App。若每个实例再执行全局 io.to(room).emit，Redis adapter 会把 A、B、C 的三次 emit 都发到全群，客户端可能收到三份。改成 io.local.to(room).emit 后，A 只通知连在 A 的客户端，B/C 同理，每个客户端正好由自己的实例通知一次。Redis adapter 仍保留给真正单源产生的 transient/global event，不需要 leader election。
Why must every listening app use io.local?
PostgreSQL already delivers the same NOTIFY to app instances A, B, and C. If each instance then calls global io.to(room).emit, the Redis adapter distributes all three emits across the cluster and a client may see three copies. With io.local.to(room).emit, A informs only A's clients and B/C do the same, so each client is notified once by its attached instance. The Redis adapter remains useful for genuinely single-origin transient/global events; no leader election is needed.
实例 B 滚动启动时，为什么不会清掉实例 A 的 presence 和运行任务？
因为恢复不再以“服务器启动”作为故障证据。A、B 各自有唯一 runtime instance ID，并在 Redis 续租 TTL heartbeat；socket session 还记录 owner instance。B 启动只注册自己，不执行全局清空。周期清理看到某个 owner heartbeat 过期后，才删除那个 owner 名下的 socket、room member socket set 和 browser presence。Running Code Agent turn/sandbox 还受 PostgreSQL fenced room lease 保护，AI streaming placeholder 受 stream-owner lease 保护；未过期的 owner 一律不恢复。Recovery 与 retention 都先抢 PostgreSQL advisory lock，同一轮只有一个实例执行。也就是说，startup 只是加入集群，lease expiry 才是 ownership 丢失的证据。
Why doesn't rolling instance B clear instance A's presence and running work?
Startup is no longer treated as evidence that existing work failed. A and B have unique runtime instance IDs and renew Redis TTL heartbeats; every socket session records its owner instance. B registers itself but performs no global reset. Periodic cleanup removes socket, room-member socket sets, and browser presence only after that owner's heartbeat expires. Running Code Agent turns and sandboxes are protected by PostgreSQL fenced room leases, while streaming AI placeholders use stream-owner leases; unexpired owners are never recovered. Recovery and retention also acquire PostgreSQL advisory locks, so one instance performs each pass. Startup merely joins the cluster; lease expiry is the evidence that ownership was lost.
权限数据库暂时不可用时，会不会把用户误判成被移除？
不会。Fast-path authorization 会把“连接自身无有效认证”和“授权依赖暂时不可用”分开。本机 socket.data 是连接生命周期内的身份权威；Redis row 丢失时从它重建。若本机身份缺失或 Redis 身份与它冲突，只让这个异常 socket 收到 registration_required 并离开 room，其他 verified socket 仍可收到完整 payload。只有 PostgreSQL membership 查询失败才属于 unavailable：服务端保留连接、不发完整 payload，只发 head-only hint 并记录 metric。客户端之后调用 get_room_events / snapshot 时再次鉴权；请求层同样返回 ROOM_AUTH_UNAVAILABLE，不会把基础设施故障翻译成 ROOM_ACCESS_DENIED。
Can an authorization database outage falsely remove a user?
No. Fast-path authorization separates an invalid connection identity from an unavailable authorization dependency. Server-owned socket.data is authoritative for that connection and repairs a missing Redis row. If the local identity is absent or conflicts with Redis, only that socket receives registration_required and leaves the room; verified peers still receive the complete payload. A PostgreSQL membership failure is unavailable rather than unauthorized: connections stay joined, no complete payload is sent, a head-only hint and metric are produced, and the later GET authorizes again. Request handlers likewise return ROOM_AUTH_UNAVAILABLE instead of translating infrastructure failure into ROOM_ACCESS_DENIED.
PG listener 断线时 NOTIFY 丢了怎么办？
NOTIFY 不是队列。Listener 必须先成功重新执行 LISTEN，然后本实例才向 local sockets 发 room_sync_required { reason: "postgres_listener_reconnected" }。客户端不清空 UI，而是从当前 lastAppliedSeq 调 get_room_events；gap replay，cursor expired 或超过 500 events 则 snapshot。先 LISTEN 再反熵，才能覆盖刚才的断线窗口和反熵过程中新到的提交。
What if PG listener downtime loses NOTIFY messages?
NOTIFY is not a queue. The listener first re-establishes LISTEN, then the instance sends local room_sync_required { reason: "postgres_listener_reconnected" }. The client keeps its UI and calls get_room_events from the current lastAppliedSeq; it replays a gap or snapshots on cursor expiry / a gap over 500 events. Re-LISTEN before anti-entropy covers both the downtime window and commits arriving during recovery.
客户端落后很多页时如何快速追赶？
当前阈值是 500 events。差距不超过 500 时，按每页最多 100 events、256 KiB 从 lastAppliedSeq 连续 replay；客户端先做一次有界 probe 以识别无法 snapshot 的 deleted-room 终态 tombstone，若普通房间差距超过 500，则不应用这页中间状态，直接取 repeatable-read snapshot，再只重放 snapshot 之后的短 tail。desiredHeadSeq 只保留通知里的最大水位，所以一秒内 1042、1043、1044、1045 会合并成"追到 1045"，不会启动四个请求。
How does a far-behind client catch up quickly?
The current threshold is 500 events. For a gap of up to 500, the client replays continuously from lastAppliedSeq in pages capped at 100 events and 256 KiB. It first makes one bounded probe to detect a terminal deleted-room tombstone, which cannot use a snapshot. For an ordinary larger gap, the client discards that intermediate page, loads a repeatable-read snapshot, and replays only the short tail. desiredHeadSeq keeps the highest notified watermark, so 1042, 1043, 1044, and 1045 arriving within one second become a single goal, "catch up to 1045," instead of four requests.
一条 AI message 从发送到完成经历什么？
鉴权后，一个 PostgreSQL 事务创建 streaming placeholder、assistant_runs、placeholder room event 与 task_dispatch_outbox。App relay 用确定性 jobId=runId 把最小 job 送入 BullMQ；ack 之后还有 active-run reconciler 检查 missing/failed job。Worker 再从 PostgreSQL 读取完整 request，claim 精确 run 并得到 generation lease；续租失败会停止 owned work，旧 generation 的 chunk/终态都无效。Chunk 经 Redis transient channel 到 App，再由 App 鉴权并 io.local 推给浏览器；早于 placeholder 的 chunk 会有界缓存。Provider 返回后，Worker 先把 immutable terminal payload staged 到 run，再用一个事务投影 Message、run terminal status 与 room cost。若 staged 后崩溃，retry 只 project；若 Provider 已接受但 staging 尚未发生，接管者可能再次调用外部服务，所以调用语义是至少一次。无论哪种顺序，只有一个 generation 能提交终态，内部费用只累计一次；placeholder 已删除或被新 generation 接管时只返回 obsolete，绝不 upsert 复活。最终 room event 让所有客户端收敛。
What happens during one AI-message lifecycle?
After authorization, one PostgreSQL transaction creates the streaming placeholder, assistant_runs, placeholder room event, and task_dispatch_outbox. The App relay enqueues a minimal job with deterministic jobId=runId, while active-run reconciliation covers missing or exhausted jobs after acknowledgement. The Worker reads the complete request from PostgreSQL and claims the exact run with a generation lease; lost renewal ends owned work, and stale chunks or terminal writes are rejected. Chunks cross a Redis transient channel to Apps, which reauthorize and emit with io.local; a chunk that beats its placeholder is bounded-buffered. After the Provider returns, the Worker stages an immutable terminal payload and atomically projects Message, run terminal state, and room cost. A crash after staging retries projection only. A crash after remote acceptance but before staging may invoke the Provider again, so external execution is at-least-once; one generation still wins terminal projection and internal cost settlement. A deleted or replacement-owned placeholder is obsolete and never upserted back. The final room event converges every client.
部署重启时，为什么 queued AI 不会被 startup recovery 判死？
assistant_runs 是业务事实，App/Worker 进程内存不是。Queued run 同时拥有 pending dispatch；migration 0010 还会为 cutover 前 active run 回填 dispatch。App 重启后 relay 继续补投；dispatch 已标为 sent、但 Redis 中 job 已不存在或 failed attempts 耗尽时，advisory-locked reconciler 会按 PostgreSQL active run 修复。Worker 重启后 BullMQ 重新 claim。Running run 的旧 generation lease 过期后，新 Worker 才能取得更高 generation；finalizing run 只 project。首次切换必须先停止旧 embedded polling Worker，再启动 BullMQ Worker，避免两套 executor 重叠。
Why doesn't startup recovery kill a queued AI job during deployment?
assistant_runs is business truth; App/Worker memory is not. A queued run has pending dispatch, and migration 0010 backfills dispatch for active pre-cutover runs. App restart resumes relay. If a dispatch was acknowledged but its Redis job disappeared or exhausted attempts, the advisory-locked reconciler repairs it from the active PostgreSQL run. Worker restart lets BullMQ reclaim work. A running generation must lose its lease before a replacement claims a higher generation, while a finalizing run performs projection only. The first cutover stops the embedded polling Worker before the BullMQ Worker starts so the two executors never overlap.
Outbox 已经是 dispatched，但 BullMQ job 丢了怎么办？
只靠 relay 不够，因为 relay 只 claim pending/processing dispatch，已经 ack 的 row 不会自动回退。RoomTalk 因此增加一条很窄的反熵路径：App 每 30 秒尝试取得同名 PostgreSQL advisory lock，选出仍为 queued/running/finalizing、dispatch 已确认且至少经过 30 秒 grace period 的 run，再按有界批次检查 BullMQ。找不到 job 就用相同 jobId=runId 补建；job 为 failed，或 job 已 completed 但业务 run 尚未终态，就重置 infrastructure attempts 并放回 waiting；waiting、active、delayed 不动。多实例只有锁持有者执行，满批次用 cursor 轮转。即使检查与 Worker 完成发生竞态，新 job 也会先读 PostgreSQL，发现 run 已 terminal 后 no-op。这个设计支持 AOF 窗口丢失、Redis volume 恢复和 PostgreSQL 恢复到空队列，但没有把数据库重新变成高频 polling scheduler。
What if the outbox says dispatched but the BullMQ job is gone?
The relay alone cannot repair that state because it claims only pending or expired-processing dispatch; an acknowledged row does not move backward automatically. RoomTalk therefore adds a narrow anti-entropy loop. Every thirty seconds an App attempts the same PostgreSQL advisory lock, selects queued/running/finalizing runs whose dispatch was acknowledged beyond a thirty-second grace period, and checks BullMQ in bounded pages. A missing job is recreated with the same jobId=runId. A failed job, or a completed job whose business run is still non-terminal, has infrastructure attempts reset and returns to waiting. Waiting, active, and delayed jobs are untouched. Only the lock holder runs a pass, and a cursor rotates across full batches. A race with normal completion is safe because any replacement job first reads PostgreSQL and no-ops on a terminal run. This supports AOF-window loss, Redis-volume recovery, and restoring PostgreSQL into an empty queue without turning the database back into a high-frequency scheduler.
AI Provider 调用能做到 exactly-once 吗？
不能对所有 Provider 做这个承诺。能严格保证的是 RoomTalk 内部的结果边界：每次执行有 generation/owner lease；旧 generation 的 chunk 和 terminal write 无效；terminal projection 锁定同一条 assistant_runs，在一个 PostgreSQL 事务中更新 Message、run 与 room cost，所以被接受的终态和内部费用只发生一次。如果 Worker 在 terminal payload staged 之后退出，重试只 project，不再调用 Provider。真正无法完全观测的是“Provider 已接受请求，但响应或 staging 前进程退出”这一小段窗口，新 generation 为了完成用户请求可能再次调用 Provider。某个 Provider 若提供可靠的 idempotency key，可以按集成收紧；通用契约应诚实地写成 Provider 至少一次、内部终态 exactly-once。
Can an AI Provider call be exactly once?
Not as a universal guarantee. RoomTalk can guarantee its own result boundary: each execution has a generation/owner lease, stale chunks and terminal writes are rejected, and terminal projection locks one assistant_runs row before updating Message, run, and room cost in one PostgreSQL transaction. That makes the accepted terminal result and internal cost settlement exactly once. If the Worker dies after terminal staging, retry performs projection only and does not call the Provider. The unobservable window is remote acceptance followed by process loss before a response is staged; a replacement generation may call the Provider again to finish the user request. A Provider-specific reliable idempotency key can tighten that integration, but the portable contract is at-least-once Provider invocation and exactly-once internal terminal projection.
BullMQ 并发为什么可以是 2，dispatch batch 却可以是 20？
它们不是同一种 ownership。Dispatch relay 的 batch 只把 20 个短小 runId enqueue 到 Redis，并用 fenced token 确认 PostgreSQL intent；它不提前 claim Provider execution。BullMQ Worker concurrency=2 表示一个进程同时 active 两个 job，每个 job 在真正开始时才 claim 自己的 assistant_run generation，并独立续租。等待中的 BullMQ job 还没有 run ownership，因此不会重现旧设计“先 claim 10 条、后 9 条在串行等待中 lease 过期”的问题。
Why can BullMQ concurrency be 2 while dispatch batch is 20?
They represent different ownership. A dispatch relay batch only enqueues twenty small runId values and acknowledges fenced PostgreSQL intent; it does not pre-claim Provider execution. Worker concurrency=2 means one process actively executes two jobs, and each job claims its own assistant_run generation only when processing begins, with independent renewal. Waiting BullMQ jobs own no run lease, so this does not recreate the old “claim ten, let nine expire while waiting serially” bug.
为什么 App 启动不再自动修改 PostgreSQL schema？
旧做法每次冷启动都重放一整套看似幂等的 DDL，其中包含 DROP/CREATE trigger、DROP/ADD constraint 和数据归一化 UPDATE。Advisory lock 只能防止两个实例同时执行，不能消除大表锁、重复扫描或滚动发布时 App 一边启动一边改 schema 的风险。现在 Compose 先运行一次性 migrate service；Kubernetes/AWS 对应 pre-deploy Job。它在事务锁下只执行 ledger 中缺失的 immutable migration，并保存 SHA-256 checksum。App 只调用只读 verifySchema()；缺 migration 或 checksum 不符就拒绝 readiness。也就是说，部署顺序明确变成 migrate → verify → serve。
Why no longer mutate PostgreSQL schema during App startup?
The old cold-start path replayed a large "idempotent" DDL bundle containing trigger replacement, constraint replacement, and normalization updates. An advisory lock prevented two instances from doing it concurrently, but it did not remove heavy locks, repeated scans, or the risk of schema mutation during a rolling start. Compose now runs a one-shot migrate service; Kubernetes/AWS should use a pre-deploy Job. It applies only missing immutable migrations under a transaction lock and records SHA-256 checksums. The App performs read-only verifySchema() and refuses readiness on a missing or changed migration. The lifecycle is explicitly migrate → verify → serve.
为什么不在收到早到的 ai_chunk 时直接创建一条假 placeholder？
因为 transient push 不是事实源：它可能重复、乱序，也可能对应最终回滚或失败的写入。客户端自己造 message 会引入一个没有数据库 seq、权限上下文和完整 metadata 的对象，之后还要解决"假对象"和 canonical after-image 合并。按 messageId 暂存只保留低延迟数据，不提前发明 durable state；真正 placeholder 到达才让它进入 UI，最终 after-image 负责兜底收敛。
Why not create a synthetic placeholder as soon as an early ai_chunk arrives?
A transient push is not authoritative: it may be duplicated, reordered, or refer to a write that ultimately fails. Inventing a client message would create an object without a database sequence, permission context, or complete metadata, followed by another merge problem when the canonical after-image arrives. Buffering by messageId preserves the low-latency data without inventing durable state; the real placeholder admits it into the UI, and the final after-image guarantees convergence.
room_events、task_dispatch_outbox 和 BullMQ 为什么不能合并？
room_events 是 fan-out replay：每个客户端有自己的 cursor，用来重建可见状态。task_dispatch_outbox 是跨存储桥：只保证已经 commit 的 run 最终被送到 Redis。BullMQ 是 competing-consumer scheduler：一个 Worker active 执行，并负责 retry/stalled recovery。把三者合并会让客户端 cursor、跨存储投递确认与 Worker ack 混成一种状态。真正的简化不是少两张表，而是让每个机制只回答一个问题；业务结果始终只问 assistant_runs。
Why can't room_events, task_dispatch_outbox, and BullMQ be one mechanism?
room_events is fan-out replay: each client has a cursor for reconstructing visible state. task_dispatch_outbox is a cross-store bridge: it guarantees that a committed run is eventually handed to Redis. BullMQ is a competing-consumer scheduler: one Worker actively executes, retries, and recovers stalled jobs. Combining them would mix client cursors, cross-store delivery acknowledgement, and Worker acknowledgement. The useful simplification is one question per mechanism; business outcome is always read from assistant_runs.
现在的 AI Worker 是最终模型吗？为什么没有 assistant_run_usage？
当前业务一致性模型已经收敛：assistant_runs 保存 request、status、generation/lease、immutable terminal payload、error 与 usage；BullMQ 只调度。终态 projection 锁定唯一 run，并在同一事务中更新 Message、run 与 room cost；run 离开 finalizing 后，重复 projection 是 no-op，所以不需要再用 assistant_run_usage 充当幂等凭证。Terminal payload 在消息删除后仍保留用量审计。运行模型也补上了必要的反熵：active run 会与 BullMQ missing/failed job 周期对账，Worker 用 TTL heartbeat 证明 consumer 存活，公开状态暴露 queue depth 与 oldest age。以后若产品需要发票级账本、跨房间重新结算或不可变财务分录，可以新增 usage ledger；现在加只会复制事实。继续演进的重点是告警、容量和 Provider-specific idempotency，而不是再造一套业务状态。
Is the AI Worker model final, and why is there no assistant_run_usage?
The business-consistency model is now converged: assistant_runs stores request, status, generation/lease, immutable terminal payload, error, and usage; BullMQ only schedules. Terminal projection locks one run and updates Message, run, and room cost in one transaction. After the run leaves finalizing, repeated projection is a no-op, so a separate assistant_run_usage row is unnecessary as an idempotency receipt. The terminal payload retains usage even if the Message is deleted. The operating model now also has the necessary anti-entropy: active runs reconcile against missing/failed BullMQ jobs, a TTL heartbeat proves consumer availability, and public status exposes queue depth and oldest age. A future invoice-grade ledger, cross-room recomputation, or immutable accounting product could justify another table; adding one now would duplicate facts. Further work belongs in alerting, capacity, and Provider-specific idempotency—not a second business state machine.
既然 assistant_runs 已经在 PostgreSQL，为什么还要 BullMQ？
assistant_runs 回答“这次用户请求是什么、现在处于什么业务状态、最终得到什么结果”；BullMQ 回答“哪个 Worker 什么时候执行、并发多少、失败后何时 retry、进程崩溃后如何识别 stalled job”。只用 PostgreSQL 当然也能做，但需要自己长期维护 polling、延迟调度、lock renewal、stalled detection、并发和运维保留。只用 BullMQ 又会把用户状态绑到 Redis runtime，难以与 Message/room cost 做同事务收敛。两者组合并不等于两套事实源，而是 durable aggregate 与 scheduler 的分工。
If assistant_runs already lives in PostgreSQL, why add BullMQ?
assistant_runs answers “what did the user request, what business state is it in, and what terminal result exists?” BullMQ answers “which Worker runs it, when, at what concurrency, after which backoff, and how a crashed active job becomes stalled.” PostgreSQL alone could implement this, but RoomTalk would keep maintaining polling, delayed scheduling, lock renewal, stalled detection, concurrency, and operational retention. BullMQ alone would tie user-visible state to Redis runtime and could not converge Message/run/cost in one database transaction. Together they are not two truth sources; they are a durable aggregate and a scheduler.
为什么现在一个 Redis 就够了？什么时候要拆两个？
当前单机规模下，一个 Redis 减少服务、备份、监控与故障点，BullMQ 又通过 queue name、connection 和 key namespace 与 realtime/cache 隔开。代码已经把 REDIS_URL 与 QUEUE_REDIS_URL 分开，因此这是“同实例、不同责任”，不是把接口写死。出现独立容量/可用性需求时再拆：例如 cache eviction 策略与 queue 的 noeviction 冲突、BullMQ latency 影响 Socket、队列需要独立 failover/维护窗口或 Worker 团队独立扩容。拆分只改配置和迁移 active job，不改 PostgreSQL aggregate 或客户端协议。
Why is one Redis enough now, and when should it split into two?
At the current single-host scale, one Redis reduces services, backups, monitoring, and failure points. Queue names, connections, and key namespaces still isolate BullMQ from realtime/cache. The code already separates REDIS_URL and QUEUE_REDIS_URL, so this is one instance with distinct responsibilities rather than a hard-coded coupling. Split when measured needs diverge: cache eviction conflicts with queue noeviction, BullMQ latency affects sockets, the queue needs independent failover/maintenance, or Worker capacity has separate ownership. The split changes configuration and active-job migration, not the PostgreSQL aggregate or browser protocol.
为什么 Socket 通知不再加一个 realtime outbox，也不需要 messageVersion？
Socket 通知失败不会丢事实：canonical mutation 与 immutable room event 已经同事务落库，客户端可按 room seq replay，所以再为通知做 claim/retry 只会维护第二套 delivery state。Outbox 适合"一个 Worker 必须执行的副作用"，不适合"所有客户端各自读取"的 fan-out。messageVersion 也只会重复 room seq；它只能说"变了"，room seq 则同时给出顺序和精确缺失范围。
Why is there no realtime Socket outbox or messageVersion?
A failed Socket notification does not lose a fact: the canonical mutation and immutable room event already committed together, and clients can replay by room sequence. Adding claim/retry for the notification would create a second delivery state. An outbox fits a side effect that one worker must execute, not fan-out that every client reads independently. messageVersion would also duplicate the room sequence: it only says "changed," while the sequence gives both order and the exact missing range.
现在迁移 AWS 难吗？
应用层映射已经明确：同一镜像分别运行 App 与 Worker 两个 ECS service（或 EKS deployment），PostgreSQL 到 RDS/Aurora，realtime 与 BullMQ 先共用 ElastiCache for Redis OSS，未来用 QUEUE_REDIS_URL 拆成独立 replication group；SeaweedFS object key 原样复制到 S3，E2B 不变。维护窗口迁移可以先停止旧 Worker、复制一致的 PostgreSQL/S3，再让新 queue 从空状态启动：pending dispatch 由 relay 发送，已 ack 但仍 active 的 run 由 reconciler 重建。若要求零停机并保留正在执行的 Provider call，仍要排空/迁移 active job，数据库也需要 logical replication/CDC 或 AWS DMS；room event log 只修复客户端同步，不能复制数据库。
How difficult is an AWS migration now?
The mapping is explicit: run the same image as separate App and Worker ECS services (or EKS deployments), PostgreSQL on RDS/Aurora, realtime plus BullMQ initially on ElastiCache for Redis OSS, and later split scheduling through QUEUE_REDIS_URL; copy SeaweedFS object keys unchanged to S3 and keep E2B. A maintenance cutover can stop the old Worker, copy a consistent PostgreSQL/S3 state, and start with an empty new queue: the relay sends pending dispatch, while reconciliation rebuilds acknowledged-but-active runs. A zero-downtime cutover that preserves in-flight Provider calls still needs active-job drain/migration, and the database needs logical replication/CDC or AWS DMS. Room events repair clients; they do not replicate databases.
数据库死锁怎么处理？
不能说"append-only 所以没有锁竞争"：每次可见写入还要更新该房间的 event-stream row 来分配下一个 seq，同房间并发写会在那里有意串行化。我们保持一致的 room/stream lock order，让业务 state 与 event 在同一事务提交；发生事务错误就整体回滚，只有带 idempotency key、可安全重放的调用才重试。真实 PostgreSQL 测试覆盖 concurrent writer、rollback 与连续序列。
How do you handle database deadlocks?
It would be wrong to say "append-only means no lock contention": every visible write also updates the room's event-stream row to allocate the next sequence, deliberately serializing concurrent writers for that room. We keep a consistent room/stream lock order and commit business state plus event in one transaction; transaction errors roll back everything, and only idempotency-keyed operations that are safe to replay may retry. Real PostgreSQL tests cover concurrent writers, rollback, and contiguous sequences.
如何保证消息顺序？
Canonical 消息历史按 server-assigned position 排序；客户端同步则按独立的 per-room event seq 只应用连续增量。Position 解决"消息怎么显示"，event seq 解决"客户端漏了哪次提交"，两者都不依赖设备时间。
How do you guarantee message ordering?
Canonical history is ordered by server-assigned message position, while client synchronization applies only a contiguous per-room event sequence. Position answers "how should messages display?" and event sequence answers "which committed changes did this client miss?" Neither depends on device clocks.
如果让你重新设计，会改什么？
三件事：1) 增加 RoomTalk-owned immutable workspace revision 层，把 rollback 从当前 sandbox Git 状态提升为可持久恢复的 product contract；目前 archive migration 保证连续性，但不是通用 revision history。2) 如果 E2B 提供稳定 command reconnect handle，就把可重连 daemon metadata 持久化，减少 control-plane restart 后的 cleanup/restart。3) 继续拆分 MessagePage 与 workspace orchestrator，把跨 panel session state 收敛到更明确的 store，而不是增加更多 props。
If you were to redesign, what would you change?
Three things: 1) Add a RoomTalk-owned immutable workspace revision layer so rollback becomes a durable product contract rather than current-sandbox Git state; archive migration preserves continuity but is not general revision history. 2) Persist reconnectable daemon metadata if E2B exposes stable command handles, reducing cleanup/restart after control-plane restarts. 3) Keep decomposing MessagePage and the workspace orchestrator so cross-panel session state converges on explicit stores instead of deeper prop chains.
怎么处理大量消息的性能？
三个层面：1) 存储层：消息分页加载（readMessagePageByRoom，默认 80 条一页），不一次性加载全部历史。2) 渲染层：React 列表虚拟化可以进一步优化（当前未做，是已知改进点）。3) 缓存层：Redis RoomMessageCacheStore 缓存最近消息，cache hit 避免查 PostgreSQL。
How do you handle performance with a large volume of messages?
Three levels: 1) Storage: paginated message loading (readMessagePageByRoom, 80 messages per page by default): never loads full history at once. 2) Rendering: React list virtualization could be added (not yet implemented: known improvement). 3) Caching: Redis RoomMessageCacheStore caches recent messages, cache hits avoid PostgreSQL queries.
项目里遇到最难的 bug 是什么？
移动端切后台回来后成员数翻倍。原因是 visibility change 触发重连，重连后 re-join 房间，但旧的 socket 还没断开，服务端认为是两个不同的连接。修复方法是在 updateRoomMemberCount 里按 clientId 去重，同一个 clientId 的多个 socket 只算一个成员。
What was the hardest bug you encountered?
Member count doubling after backgrounding on mobile. The cause: visibility change triggered a reconnect, which re-joined the room, but the old socket hadn't disconnected yet: the server counted two separate connections. The fix was deduplicating by clientId in updateRoomMemberCount: multiple sockets from the same clientId count as one member.

1. 受控直接切换

先做 shadow restore、真实数据/对象演练和完整 smoke，最终用一次 write freeze + final delta + DNS 切换；上线后不保留双 writer。

2. Safety Guards

The persistence smoke test rejects a database name that does not contain test or e2e. This guard reduces the chance of hitting production, but the name is still configuration input; it does not replace isolated credentials, separate networks, least privilege, or an explicit test environment.

3. 文档即代码

3. Docs as Code

server/.env.example, .env.compose.example, Compose, and the configuration reference define the runtime contract; real secrets live only in macOS Keychain.

4. Release 守门

Compose 对 PostgreSQL/S3 credential fail-fast，容器 healthcheck、loopback/public status 与真实边界 smoke 共同决定发布成功；Source push 与生产部署是两种状态。这里的 healthcheck 不是只验证端口：App readiness 会真正读取 RoomTalk 的 PostgreSQL schema、执行 Redis PING、探测对象存储 bucket，并确认 Socket adapter 的 pub/sub 两个 Redis client 都处于 connected ready，而不是只检查 adapter 对象已经创建。AI Worker 还每 5 秒续租一个默认 20 秒 TTL 的共享 heartbeat；公开 status 同时输出 waiting、active、delayed、failed 和 oldest queued time，避免“Redis 在线、consumer 已死”仍显示 online。

4. Release Guardrails

Compose fails fast on missing PostgreSQL/S3 credentials, while container health checks, loopback/public status, and boundary smokes define release success. Source push and production deployment are separate states. The App health check does more than open a port: readiness reads the RoomTalk PostgreSQL schema, executes Redis PING, probes the object-storage bucket, and requires both Redis pub/sub clients behind the Socket adapter to be connected—not merely constructed. The AI Worker also renews a shared heartbeat every five seconds with a twenty-second default TTL. Public status reports waiting, active, delayed, failed, and oldest queued time, so “Redis is up but no consumer is alive” cannot remain online.

为什么要分 liveness、readiness 和 degraded？

Liveness 只回答“这个 Node 进程还能不能处理 HTTP”。如果 PostgreSQL 短暂不可用，不应该让 Kubernetes 反复重启一个本来正常的 App 进程，因此 /api/health/live 不访问下游。Readiness 回答“现在把用户流量送进来是否安全”，所以 /api/health/ready 与 /api/status 必须检查所有关键依赖；失败时返回 HTTP 503 和 status: "degraded"。

最重要的是保留领域含义：rooms: 0 表示数据库查询成功，而且确实没有房间；数据库查询失败则是 rooms: null，并把 dependencies.database 标为 unavailable。不能为了让 status 看起来“在线”而把 infrastructure error 转换成合法业务值。Compose 用 readiness 判断 App 健康；迁移 AWS 后，load balancer/readiness 停止导流，liveness 只在进程自身失效时触发重启。

Queue/Worker 采用另一种 degraded 语义：只要 PostgreSQL 可写，App 仍能可靠接受请求并留下 dispatch intent，所以不会因为 Worker 短暂重启就把整个 HTTP 实例摘流；但 status 会变成 degraded，并标明 assistantWorker: unavailable。这样既不丢请求，也不会把“任务没人消费”伪装成完全在线。

Why separate liveness, readiness, and degraded?

Liveness asks only whether the Node process can still handle HTTP. A temporary PostgreSQL outage should not make Kubernetes restart an otherwise healthy App process, so /api/health/live has no downstream checks. Readiness asks whether it is safe to send user traffic now. Therefore /api/health/ready and /api/status check every required dependency and return HTTP 503 with status: "degraded" when one fails.

The domain distinction matters: rooms: 0 means the query succeeded and the database really contains no rooms. A failed query produces rooms: null and dependencies.database: "unavailable". Infrastructure errors must not be converted into valid business values merely to keep a status page green. Compose consumes readiness; on AWS, readiness removes a replica from traffic while liveness restarts it only when the process itself has failed.

Queue and Worker failure use a different degraded mode. As long as PostgreSQL remains writable, the App can safely accept a request and preserve dispatch intent, so a brief Worker restart does not remove the whole HTTP replica from traffic. Status still becomes degraded with assistantWorker: unavailable. The system neither loses the request nor pretends that background execution is healthy.

迁移脚本可以反复跑，room upsert + message 按 ID 去重。

Migration scripts can be run repeatedly: room upsert + message dedup by ID.

当前 Git + archive migration 保证 live workspace 连续性，但还不是 immutable revision/rollback 产品层。面试时要清楚区分已经实现的能力、恢复机制和下一步架构，比把所有能力都说成完成更可信。

7. Explicit Boundaries

Git plus archive migration preserves live workspace continuity, but it is not yet an immutable revision/rollback product layer. Clearly separating shipped behavior, recovery mechanisms, and next architecture steps is more credible than claiming everything is complete.
