1. CompositeRoomStore — 双存储分层1. CompositeRoomStore — Dual-Store Layering

这是整个后端最核心的抽象。一个 CompositeRoomStore 组合了三个子存储：

This is the most important abstraction in the backend. A single CompositeRoomStore composes three sub-stores:

  • DurableRoomStore：持久数据（房间、消息、成员、媒体资产、认证）。可以是 Redis 或 PostgreSQL，运行时通过环境变量切换。
  • RealtimeRoomStore：始终是 Redis。管理 socket session、在线成员、临时状态。服务重启后自动重建。
  • RoomMessageCacheStore：可选的 Redis TTL 缓存，在 PostgreSQL 模式下减少数据库读压力。
  • DurableRoomStore: Persistent data (rooms, messages, members, media assets, auth). Can be Redis or PostgreSQL, switched at runtime via environment variable.
  • RealtimeRoomStore: Always Redis. Manages socket sessions, online members, ephemeral state. Rebuilt automatically on restart.
  • RoomMessageCacheStore: Optional Redis TTL cache that reduces database read pressure in PostgreSQL mode.
我们把存储拆成"持久"和"实时"两层。持久层可以在 Redis 和 PostgreSQL 之间切换——开发用 Redis 快速迭代，生产切到 PostgreSQL 保证数据安全。实时层始终用 Redis 因为它本来就是内存存储，做 session 和 presence 正好。中间加了一层消息缓存，cache miss 才查 PostgreSQL，命中率在 90%+ 因为聊天场景下用户大概率读最近的消息。
We split storage into "durable" and "realtime" layers. The durable layer can switch between Redis and PostgreSQL — Redis for fast iteration in development, PostgreSQL for data safety in production. The realtime layer always uses Redis since it's an in-memory store, perfect for sessions and presence. We added a message cache layer in between — only queries PostgreSQL on cache miss, with 90%+ hit rate because chat users mostly read recent messages.

追问：sticky session 为什么必须？

Socket.IO 握手分两步：先发几个 HTTP long-polling 请求，再升级到 WebSocket。如果这几个 HTTP 请求被负载均衡分到不同实例，握手就断了。所以 ALB 必须开 stickiness，保证同一个客户端的请求始终到同一台机器。握手完成后 WebSocket 是长连接，自然不会飘。

Follow-up: Why are sticky sessions required?

Socket.IO handshake has two phases: first a few HTTP long-polling requests, then an upgrade to WebSocket. If those HTTP requests hit different instances, the handshake breaks. So the ALB must enable stickiness to route the same client to the same machine. Once the WebSocket connection is established, it's a persistent connection and won't drift.

追问：为什么不用 SSE？

已经有 Socket.IO 长连接了，再开 SSE 是多余的连接。Socket.IO 事件模型可以区分不同房间、不同用户的 AI 流，比 SSE 灵活。而且 AI 流需要和消息历史、状态同步走同一个通道，不适合单独开一个 HTTP 流。

Follow-up: Why not use SSE?

We already have a Socket.IO persistent connection — opening another SSE connection would be redundant. Socket.IO's event model can distinguish AI streams by room and user, which is more flexible than SSE. Plus, AI streams need to share the same channel as message history and state updates — a separate HTTP stream doesn't fit.

4. 媒体上传流程 — Presigned URL4. Media Upload Flow — Presigned URL

Client Server S3 │ │ │ │── request upload URL ──▶│ │ │ │── generate presigned ──▶│ │◀── presigned PUT URL ──│ │ │ │ │ │──── PUT file directly ─────────────────────────▶ │ │ │ │ │── confirm upload ──────▶│ │ │ │── create MediaAsset ──▶ │ │◀── message with asset ─│ │
文件不经过我们的服务器。客户端先向服务器请求一个临时上传 URL（presigned URL，有效期 15 分钟），然后直接上传到 S3。上传完成后通知服务器，服务器创建 metadata 记录。下载也是类似的，服务器生成临时读取 URL 返回给客户端。这样服务器不处理大文件流量，带宽和 CPU 都省了。
Files never pass through our server. The client requests a temporary upload URL (presigned URL, valid for 15 minutes) from the server, then uploads directly to S3. After upload, it notifies the server, which creates a metadata record. Downloads work similarly — the server generates a temporary read URL. This way the server handles zero file traffic, saving both bandwidth and CPU.

追问：presigned URL 安全吗？

URL 本身包含签名，只对特定 bucket/key 有效，有过期时间。泄露了也只能上传到这一个位置，不能读其他文件。而且我们的 bucket 设置了 block all public access，没有 presigned URL 什么都看不到。

Follow-up: Are presigned URLs secure?

The URL itself contains a cryptographic signature, valid only for a specific bucket/key and with an expiration time. Even if leaked, it can only upload to that one location — it can't read other files. Our bucket has "block all public access" enabled, so nothing is visible without a presigned URL.

const getCachedMediaObjectUrl = async ({ assetId, url, kind, byteSize }) => {
  const key = bodyCacheKey(assetId);   // 用 assetId 做 key，不用变化的 URL

  // 第 0 层：并发请求去重（同一资源复用 Promise）
  const existing = inFlightBodyUrls.get(key);
  if (existing) return existing;

  const request = (async () => {
    try {
      // 第 1 层：内存中的 Object URL（最快，无 IO）
      const cachedUrl = await getBlobObjectUrlFromCache(MEDIA_BODY_CACHE_NAME, key);
      if (cachedUrl) return cachedUrl;

      // 第 2 层：Cache API blob → 生成 Object URL
      const response = await fetch(url, { cache: "force-cache" });
      const blob = await response.blob();
      await putBlobInCache(MEDIA_BODY_CACHE_NAME, key, blob, mimeType, MAX_BODY_CACHE_BYTES);
      return rememberObjectUrl(key, URL.createObjectURL(blob));
    } finally {
      inFlightBodyUrls.delete(key);    // 无论成功失败都清除
    }
  })();

  inFlightBodyUrls.set(key, request);  // 注册 in-flight
  return request;
};

    .readCachedRoomMessages(roomId, cacheVersion);

      roomId, messages, cacheVersion

难点 13：AI 上下文预算管理（CJK 感知）Challenge 13: AI Context Budget Management (CJK-Aware)

问题：AI 请求附带聊天历史上下文，太长会超模型 token 限制。难点在于准确估算——中文 1 字 ≈ 1 token，英文 1 词 ≈ 1 token，混合内容需分别计算。

Problem: AI requests include chat history as context, but exceeding the model's token limit causes errors. The challenge is accurate estimation — Chinese uses ~1 token per character, English ~1 token per word, and mixed content needs separate calculation.

// aiHistory.ts — CJK 感知的 token 估算
const estimateTextTokens = (text: string): number => {
  const cjkCount = text.match(CJK_CHARACTER_PATTERN)?.length ?? 0;
  const nonCjkText = text.replace(CJK_CHARACTER_PATTERN, '');
  return Math.max(1, cjkCount + Math.ceil(nonCjkText.length / 4));
  // CJK: 1 字符 ≈ 1 token；拉丁: 4 字符 ≈ 1 token
};

三层筛选（从粗到细）：

Three-Layer Selection (coarse to fine):

// aiHistory.ts — selectAIHistory
export function selectAIHistory(fullHistory, options) {
  let history = fullHistory;

  // 第 1 层：语义截断（重试/编辑时截到该消息之前）
  if (retryForMessageId) {
    const idx = history.findIndex(m => m.id === retryForMessageId);
    if (idx !== -1) history = history.slice(0, idx);
  }

  // 第 2 层：消息条数限制（最近 N 条）
  if (history.length > maxContextMessages)
    history = history.slice(-maxContextMessages);

  // 第 3 层：token 预算（从最新往前贪心累加）
  const selected = [];
  let tokenCount = 0;
  for (let i = history.length - 1; i >= 0; i--) {
    const msgTokens = estimateMessageTokens(history[i]);
    if (selected.length > 0 && tokenCount + msgTokens > maxContextTokens)
      break;
    selected.push(history[i]);
    tokenCount += msgTokens;
  }
  return { contextMessages: selected.reverse(), contextTokenEstimate: tokenCount };
}
AI 上下文管理分三层筛选。第一层是语义截断——用户重试或编辑消息时，只取该消息之前的历史。第二层是条数限制。第三层是 token 预算，从最新消息往前贪心累加。token 估算对 CJK 和拉丁字符分别计算：中文 1 字约 1 token，英文约 4 字符 1 token。这个估算偏保守，宁可少发一条也不超限。
AI context management uses three-layer selection. Layer 1 is semantic truncation — when retrying or editing, only history before that message is included. Layer 2 is a message count limit. Layer 3 is token budget — greedy backward accumulation from the newest message. Token estimation calculates CJK and Latin characters separately: ~1 token per CJK character, ~4 Latin characters per token. The estimate is intentionally conservative — better to include one fewer message than to exceed the limit.

为什么用 Socket.IO 而不是原生 WebSocket？
Socket.IO 提供了开箱即用的自动重连、房间/命名空间、ack 回调、Redis adapter 多实例广播。原生 WebSocket 这些都要自己实现。对于聊天场景，Socket.IO 省了大量样板代码。
Why Socket.IO instead of raw WebSocket?
Socket.IO provides auto-reconnection, rooms/namespaces, ack callbacks, and Redis adapter for multi-instance broadcast out of the box. With raw WebSocket, you'd implement all of these yourself. For a chat scenario, Socket.IO saves enormous amounts of boilerplate.
为什么不用微服务架构？
当前规模不需要。单体的开发、部署、调试效率远高于微服务。如果将来 AI 处理成为瓶颈，可以把 AI worker 拆出来用队列解耦，但现在拆了是过度设计。
Why not a microservices architecture?
The current scale doesn't justify it. A monolith is far more efficient for development, deployment, and debugging. If AI processing becomes a bottleneck in the future, we can extract an AI worker and decouple with queues — but splitting now would be over-engineering.
Redis 挂了怎么办？
实时状态（在线成员、session）丢失，用户需要重连，服务端重建。持久数据在 PostgreSQL，不受影响。消息缓存丢失只是多查一次 PostgreSQL，有性能影响但不丢数据。
What happens if Redis goes down?
Realtime state (online members, sessions) is lost — users need to reconnect, and the server rebuilds it. Persistent data lives in PostgreSQL and is unaffected. Message cache loss just means an extra PostgreSQL query — performance impact but no data loss.
数据库死锁怎么处理？
当前消息写入是 append-only（INSERT），不存在行锁竞争。房间设置更新用 room_version 行级版本号 + UPDATE ... RETURNING 原子操作，失败重试。PostgreSQL 的 MVCC 天然避免了读写阻塞。
How do you handle database deadlocks?
Message writes are append-only (INSERT), so there's no row-lock contention. Room settings use room_version row-level versioning + UPDATE ... RETURNING atomic operations with retry on failure. PostgreSQL's MVCC naturally avoids read-write blocking.
如何保证消息顺序？
每条消息有 server-side timestamp 和 position（单调递增）。客户端按 position 排序显示。不依赖客户端时间戳，因为多设备间的时钟可能不同步。
How do you guarantee message ordering?
Every message has a server-side timestamp and a monotonically increasing position. The client sorts by position. We don't rely on client timestamps because clocks across devices can be out of sync.
如果让你重新设计，会改什么？
三件事：1) 一开始就用 PostgreSQL 而不是 Redis 做持久化，省去后来的迁移工作。2) Socket 事件的错误返回用结构化错误码，而不是字符串匹配（目前还有少量 regex 匹配遗留）。3) 前端状态管理用 Zustand 或类似库，MessagePage 承担了太多状态编排，组件间共享状态靠 props 传递层级太深。
If you were to redesign, what would you change?
Three things: 1) Start with PostgreSQL instead of Redis for persistence — would've avoided the migration effort. 2) Use structured error codes for socket event errors instead of string matching (some legacy regex matching remains). 3) Use Zustand or similar for frontend state management — MessagePage handles too much state orchestration, and shared state passes through too many prop layers.
怎么处理大量消息的性能？
三个层面：1) 存储层：消息分页加载（readMessagePageByRoom，默认 80 条一页），不一次性加载全部历史。2) 渲染层：React 列表虚拟化可以进一步优化（当前未做，是已知改进点）。3) 缓存层：Redis RoomMessageCacheStore 缓存最近消息，cache hit 避免查 PostgreSQL。
How do you handle performance with a large volume of messages?
Three levels: 1) Storage: paginated message loading (readMessagePageByRoom, 80 messages per page by default) — never loads full history at once. 2) Rendering: React list virtualization could be added (not yet implemented — known improvement). 3) Caching: Redis RoomMessageCacheStore caches recent messages, cache hits avoid PostgreSQL queries.
项目里遇到最难的 bug 是什么？
移动端切后台回来后成员数翻倍。原因是 visibility change 触发重连，重连后 re-join 房间，但旧的 socket 还没断开，服务端认为是两个不同的连接。修复方法是在 updateRoomMemberCount 里按 clientId 去重，同一个 clientId 的多个 socket 只算一个成员。
What was the hardest bug you encountered?
Member count doubling after backgrounding on mobile. The cause: visibility change triggered a reconnect, which re-joined the room, but the old socket hadn't disconnected yet — the server counted two separate connections. The fix was deduplicating by clientId in updateRoomMemberCount — multiple sockets from the same clientId count as one member.
