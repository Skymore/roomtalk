# RoomTalk 部署指南

[English](DeploymentGuide.md)

更新：2026-07-12
生产：[https://room.ruit.me/](https://room.ruit.me/)

## 生产形态

RoomTalk 当前生产使用：

- `dfw` 的 Fly.io 应用 `message-system` 承载 Node control plane；
- `fly.toml` 声明的 shared 1-vCPU、1024 MB 机器；
- Supabase PostgreSQL 作为 durable source of truth；
- Upstash Redis 承载 Socket.IO、presence、session、pub/sub、counter 和有界 message cache；
- Tigris/S3-compatible object storage 保存私有媒体和已发布 static artifact；
- E2B 承载 room-scoped Code Agent execution sandbox；
- 固定 E2B artifact 内置可复用 daemon 和 Codex app-server backend。

无流量时应用可以 auto-stop，由 Fly Proxy auto-start。首个请求的 cold-start latency 与应用健康需要分开判断。

## 发布归属

`master` 是 release branch，`.github/workflows/fly-deploy.yml` 拥有应用部署。

Workflow 会：

1. 每两小时的第 23 分钟运行，或被手动 dispatch；
2. 比较 `master` 与最近成功 workflow run；
3. 没有相关改动时跳过；
4. comparison 不完整或非线性时保守构建；
5. 校验必需 secret、翻译、client/server production build；
6. 构建根目录 multi-stage Docker image；
7. 部署到 Fly 并验证。

不要手动执行 `fly deploy`。Git push 本身不能证明生产已部署，因为 workflow 不由 push 触发。

## 常规发布

1. 按改动风险完成验证。
2. 提交并推送已完成改动到 `origin/master`。
3. 判断定时 workflow 是否足够，或是否需要立即 rollout。
4. 需要立即 rollout 时，在 GitHub Actions dispatch `fly-deploy.yml`。
5. 验证 workflow conclusion 和已部署 commit/image。
6. 验证 Fly machine state 和 public status endpoint。

只读检查：

```bash
gh run list --repo Skymore/roomtalk --workflow fly-deploy.yml --branch master --limit 5
fly status -a message-system
curl -fsS https://room.ruit.me/api/status
```

Status 应返回 `status: "online"`、预期 `persistenceStore` 和合理的 room count。

## 首次环境准备

常规 workflow 假设 Fly 应用和外部服务已存在。新环境需要：

1. 创建 Fly 应用，并在 `fly.toml` 设置目标 app/region。
2. 准备 PostgreSQL、Redis 和 S3-compatible object storage。
3. 创建专用 PostgreSQL application role。
4. 添加 deployment workflow 需要的 GitHub Actions/Fly 凭据。
5. 通过平台 secret manager 添加应用 secret。
6. 启用生产 Code Agent 房间前，构建并发布固定 E2B template。
7. Dispatch workflow，完成下方验证清单。

不要把生产 serving machine 当作通用 migration host。

## 配置分组

完整 operator-facing 清单见 [docs/configuration.zh.md](docs/configuration.zh.md)。重要生产分组：

| 范围 | 代表变量 |
| --- | --- |
| HTTP/origin | `NODE_ENV`, `PORT`, `CLIENT_URL`, `CLIENT_URLS` |
| Store | `REDIS_URL`, `PERSISTENCE_STORE`, `DATABASE_URL`, PostgreSQL TLS/CA, message-cache limit |
| 媒体 | bucket、endpoint、region、S3 credential |
| Chat AI | provider key、default model、context limit |
| 可选产品服务 | Google OAuth、AssemblyAI、Web Push |
| Code Agent | enablement/allowlist、mode、daemon/backend、E2B pin、scoped capability secret |
| 用户自有 connection | Codex/GitHub enablement 和 encryption key |

`CLIENT_URL` 用于 RoomTalk canonical browser address。需要接受其他 browser origin 的部署使用逗号分隔 `CLIENT_URLS`。仓库文档和生成示例只使用 canonical RoomTalk address。

修改 Fly secret 会滚动或重启 machine，每次修改后都要验证健康。

## 存储模型与 PostgreSQL 切换

RoomTalk 只有一种 serving model：`PERSISTENCE_STORE=postgres`。PostgreSQL 负责 durable canonical state 与 room-event replay；Redis 仍必需，但只保存可重建的实时协调和有界缓存。Runtime 会拒绝把 Redis 当成 durable authority。

PostgreSQL dump/restore、验证、写入冻结和回滚边界以[可迁移部署与直接切换设计](docs/room-event-sync-portable-deployment.zh.md)为准。[旧导入 runbook](docs/postgres-rollout-runbook.zh.md)只用于把旧 Redis durable snapshot 导入 PostgreSQL。回滚是恢复或 failover PostgreSQL，不是把 durable authority 切回 Redis。

## Code Agent / E2B 发布合约

Fly 部署 control plane，不会重建已存在的 E2B template。Runner、daemon、tool、prompt、sandbox Dockerfile、dependency lock 或 code-agent-engine 改动需要：

1. 已提交并推送的源码；
2. 根据情况更新 runner version/lock/source ref；
3. 新 artifact version 和 E2B template；
4. 更新生产 `CODE_AGENT_E2B_TEMPLATE_ID`、`CODE_AGENT_ARTIFACT_VERSION` 和 `CODE_AGENT_SOURCE_REF`；
5. 真实 E2B smoke 或等价直接验证。

详见 [artifact 合约](docs/code-agent-sandbox-artifact.zh.md)。

## 验证清单

### Control plane

- GitHub Actions 对预期 `master` commit 执行成功。
- Fly 报告预期 machine image/version 和 `started` state。
- `/api/status` 在线并报告预期 persistence mode。
- PostgreSQL、Redis、object storage 和 socket adapter 初始化没有错误。

### 用户流程

- 打开 canonical application URL。
- 注册/加入 room 并刷新。
- 发送文本消息，确认刷新后仍存在。
- 媒体配置变化时验证 private media upload/read。
- AI 配置变化时确认 streaming 只生成一个 durable final response。
- Socket 或 client state 改动时验证 mobile/reconnect。

### Code Agent

改动涉及此边界时：

- 创建或打开有权限的 Code Agent room；
- 验证 sandbox create/reconnect 和 artifact metadata；
- 运行受影响 Coco/Codex backend 和 mode；
- 按需验证 ordered text/tool event、usage、terminal/preview/workspace access；
- lifecycle 变化时验证 pause/resume 或 replacement。

## 回滚

### 应用 rollout

通过受控 Fly/GitHub workflow 重新部署已知正常的 application image。保留日志和失败 image/commit 用于分析。除非已测量数据分歧，不要把 application rollback 与 database-mode rollback 合并。

### Secret/配置改动

通过 Fly 恢复上一个已知正常值，等待 rolling update，再重复 health check。

### E2B artifact

把生产 pin 指回上一组已验证 template/artifact/source ref，然后验证新 sandbox create。已存 sandbox 可能仍带有旧 workspace/runtime state，需要显式 lifecycle 处理。

### Database

遵循 PostgreSQL runbook。纯配置回滚只在写入冻结、尚无 PostgreSQL-only write 的切换窗口内安全。

## 运维

```bash
fly status -a message-system
fly logs -a message-system
fly machine list -a message-system
```

需要 shell syntax 时使用 `fly ssh console -C 'sh -lc "..."'`。不要把 secret value 打印或复制到 issue、日志或文档。

Scaling 改动必须提交到 `fly.toml`，使仓库保持事实源。当前声明为 one shared CPU 和 1024 MB；不要把 ad-hoc console scaling 写成期望生产状态。
