# RoomTalk 部署指南

[English](DeploymentGuide.md)

状态：当前生产 runbook
更新：2026-07-20
生产：[https://room.ruit.me/](https://room.ruit.me/)

## 生产形态

RoomTalk 当前在 MacBook 上长期运行五个 Docker Compose 服务：

- `app`：根 multi-stage application image，Node/Express/Socket.IO 只绑定 loopback `3012`；
- `postgres`：PostgreSQL 17，是唯一 durable serving authority，使用 `postgres_data` Docker volume；
- `redis`：Redis 7，承担 Socket.IO、presence、session 与有界 cache，刻意关闭持久化；
- `object-storage`：SeaweedFS 4.29，暴露 S3-compatible API，数据位于 `runtime/object-storage`；
- `cloudflared`：出站 Cloudflare Tunnel，在不开放宿主机入站端口的情况下提供公网 DNS/TLS。

`room.ruit.me` 是 canonical host，`roomtalk.ruit.me` 是兼容入口，`roomtalk-objects.ruit.me` 承载浏览器预签名对象传输。每房间 Code Agent sandbox 仍由外部 E2B execution plane 提供。

旧 Fly app、Supabase、Tigris 与 Upstash 只保留为回滚源。Fly 已暂停，定时 GitHub Actions workflow 已手工禁用；这些旧服务都不接收生产写入。

## 发布归属

`master` 是 source release branch，但 Git push 不会部署 Mac。生产 checkout 与 `scripts/local-production.mjs` 拥有应用 rollout。脚本从 macOS Keychain item `roomtalk-production-env` 读取应用环境，为本次 Compose 调用生成 mode `0600` 的临时 env file，并在命令结束后删除。

应用镜像与固定 E2B artifact 是两条独立发布链路。普通 app rollout 不会重建已有 E2B template。

## 常规应用发布

1. 按改动的真实故障模式完成验证。
2. 提交并推送完成的改动到 `origin/master`。
3. 确认生产 checkout 位于目标 commit，且没有无关 tracked change：

   ```bash
   git fetch origin master
   git status --short --branch
   test "$(git rev-parse HEAD)" = "$(git rev-parse origin/master)"
   ```

4. 构建并 reconcile 生产 stack：

   ```bash
   node scripts/local-production.mjs --profile edge up -d --build
   ```

5. 验证容器与本地/公网健康：

   ```bash
   node scripts/local-production.mjs --profile edge ps
   curl -fsS http://127.0.0.1:3012/api/status
   curl -fsS https://room.ruit.me/api/status
   ```

6. 按风险运行用户 smoke。Room event、media、OAuth/connection 与 E2B 改动必须验证真实边界。

纯文档 commit 不需要重建生产镜像。

## 首次准备宿主机

1. 安装 Docker Desktop；生产 session 期间 Mac 保持接电并禁用自动睡眠。
2. 复制 `.env.compose.example` 为 ignored `.env.compose`，生成彼此独立的 PostgreSQL 与本地 S3 credential。
3. 把完整应用环境作为 JSON object 存入 macOS Keychain item `roomtalk-production-env`，不得提交或打印。
4. 为专用 tunnel 创建 ignored `runtime/cloudflared/config.yml` 与 `runtime/cloudflared/credentials.json`。
5. 恢复成对的 PostgreSQL custom archive 与 SeaweedFS snapshot，或者初始化空环境。
6. 启用 Code Agent 房间前，验证固定 E2B template/artifact/source-ref 组合。
7. 使用 `node scripts/local-production.mjs --profile edge up -d --build` 启动，并完成下方验证清单。

完整 operator-facing 变量清单见 [docs/configuration.zh.md](docs/configuration.zh.md)。

## Durable State 与 Room Event Sync

PostgreSQL 保存 canonical rooms、messages、members、turns、auth/account、media metadata、`room_event_streams`、`room_events` 与 `outbox_events`。Redis 可清空后重新预热，不会丢失业务事实。

`room_events` 是所有获授权客户端读取的每房间有界 replay changelog，不是完整 Event Sourcing，也不是 Worker queue。`outbox_events` 是独立的单 Worker claim/lease/retry 机制。默认每房间保留七天、最多 10,000 条事件，并每小时清理旧连续前缀；详见[房间事件同步与可迁移部署](docs/room-event-sync-portable-deployment.zh.md)。

Event log 用于漏掉 Socket.IO 通知后的客户端同步修复，不能替代 PostgreSQL backup、WAL/CDC 或数据库复制。

## 备份与恢复

只在已通知的维护窗口执行成对备份：

```bash
node scripts/backup-local-production.mjs
```

此命令没有 help 或 dry-run 模式。运行后会立即停止 `cloudflared`、`app` 与 `object-storage`，在 `backups/` 生成 PostgreSQL custom archive 和同一时间点的 SeaweedFS tarball，并通过 `finally` 路径重新启动服务。

每次备份后：

- 确认五个服务健康，公网 `/api/status` online；
- 数据库 dump 与 object snapshot 必须作为同一 timestamp pair 保存；
- 把加密 artifact 复制到 Mac 之外；
- 定期恢复到隔离目标并比较数据库与对象 count。

只有本机 `backups/` 不构成灾难恢复。

## Code Agent / E2B 发布合约

Runner、daemon、tool、prompt、sandbox Dockerfile/dependency lock 或固定 code-agent-engine 发生变化时，需要：

1. 已提交并推送的源码；
2. 根据范围更新 runner/artifact version 与 source ref；
3. 构建并发布新的 E2B template；
4. Keychain 环境中的 `CODE_AGENT_E2B_TEMPLATE_ID`、`CODE_AGENT_ARTIFACT_VERSION` 与 `CODE_AGENT_SOURCE_REF` 对齐；
5. 真实 E2B smoke 或等价直接验证；
6. 只有 control-plane 配置或源码也变化时才需要重启 app。

详见 [artifact 合约](docs/code-agent-sandbox-artifact.zh.md)。

## 验证清单

### Control plane

- 五个 Compose 服务都为 healthy/running。
- Loopback 与公网 `/api/status` 返回 `status: "online"`、`persistenceStore: "postgres"`、Redis connected、media configured、socket adapter ready。
- `room.ruit.me` 与 `roomtalk.ruit.me` 提供目标应用；对象域名只接受签名对象操作。
- PostgreSQL schema/event listener、Redis、object storage、worker 与 Socket.IO 启动没有错误。

### 用户与同步流程

- 打开或加入房间，发送文本并刷新，消息仍然存在。
- 第二客户端收到 wake-up，并通过 `get_room_events` 收敛。
- 相关改动需要覆盖 offline replay、`CURSOR_EXPIRED`/resnapshot、edit/delete/clear 与 deleted-room tombstone。
- Storage/edge 变化时验证 presigned media PUT/GET。
- Google/GitHub/Codex connection 与 E2B 边界变化时分别验证。

## 回滚

### 应用镜像

把生产 checkout 移到已知正常 commit，用 `scripts/local-production.mjs` 重建，并重复健康与用户流程检查。保留失败 commit 和日志用于分析。

### 配置

恢复上一份已知正常的 Keychain JSON object，reconcile stack 并重复健康检查。排障时不要打印环境。

### 数据库与对象

必须从同一 timestamp maintenance pair 恢复 PostgreSQL 与 SeaweedFS。只恢复一侧会使 media metadata 与 object bytes 不一致。

### 旧云端目标

Mac 已接受写入后，不能只重新启用 Fly 或切 DNS。先把 PostgreSQL 与 S3 增量协调到目标，停止或 gate 当前 writer，验证恢复目标，然后才能切流量。

## 运维

只读检查：

```bash
node scripts/local-production.mjs --profile edge ps
node scripts/local-production.mjs --profile edge logs --tail=200 app
node scripts/local-production.mjs --profile edge logs --tail=200 cloudflared
curl -fsS https://room.ruit.me/api/status
```

长期容器使用 Docker JSON 日志轮转：单文件 10 MB、保留五份。数据库中的 observability、turn、event 与 outbox record 不受这个 process-log 上限影响。

迁移 AWS 时保持同一镜像与合约：app 映射到 ECS Fargate 或 EKS，PostgreSQL 到 RDS/Aurora，Redis 到 ElastiCache，SeaweedFS objects 到 S3。遵循[顶层 AWS 可迁移性章节](README.zh.md#aws-可迁移性)的受控切换；真正零停机还需要 PostgreSQL logical replication/CDC 或 AWS DMS。
